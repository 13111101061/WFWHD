### 1.3 已发现的问题

#### 问题 1：ProviderCatalog 是死依赖

`TtsSynthesisService` 构造函数注入了 `voiceCatalog`（ProviderCatalog 实例），但合成链路中从未使用。VoiceResolver 通过 VoiceRegistry 查找 provider，不走 ProviderCatalog。

```javascript
// TtsSynthesisService 构造函数
constructor({ ttsProvider, voiceCatalog, validator, ... }) {
  // voiceCatalog 被保存但合成路径中从未调用
}
```

#### 问题 2：ProviderManagementService 有重复方法

`_checkCredentialStatus(providerKey)` 和 `_checkServiceCredentialStatus(providerKey, serviceType)` 功能高度相似，前者只检查 provider 级别凭证，后者检查 service 级别。`checkServiceAvailability` 只使用了后者，前者无调用者。

```javascript
// 死代码 - 从未被调用
_checkCredentialStatus(providerKey) {
  const configured = this._credentials.isConfigured(providerKey);
  return { configured, reason: configured ? 'ok' : 'not_configured' };
}

// 实际使用中
_checkServiceCredentialStatus(providerKey, serviceType) {
  const configured = this._credentials.isServiceAvailable(providerKey, serviceType);
  return { configured, reason: configured ? 'ok' : 'not_configured' };
}
```

#### 问题 3：getAllServices 的 options 参数被静默丢弃

```javascript
// ProviderManagementService — options 被透传
getAllServices(options) { return this._registry.getAll(); }

// ProviderRegistry.getAll — 不接受参数，options 被忽略
getAll() { return Array.from(this._descriptors.values()); }
```

#### 问题 4：serviceType 手动切割容易出错

```javascript
// ProviderRegistry.initialize() — 从 canonical key 硬切
const serviceType = key.slice(providerKey.length + 1);
// "aliyun_cosyvoice" → providerKey="aliyun" → "cosyvoice" ✓
// 但如果 providerKey 拼写不匹配，会切出错误结果
```

#### 问题 5：Provider 粒度健康度缺失

ExecutionPolicy 的 circuit breaker 是 per-serviceKey 的。同一 provider 的多个 service（如 `aliyun_cosyvoice` + `aliyun_qwen_http`）如果共享底层连接，故障会互相影响但无法感知。

```
  ProviderCatalog         → 查询（给前端）
  ProviderManagementService → 查询 + 凭证 + 适配器（给合成链路）
  ProviderRegistry         → 底层存储

问题：TtsSynthesisService 注入了 ProviderCatalog 但从不用它
```

### 方案 2：Provider 调用可观测性

**目标**：每次合成调用产生结构化指标记录，支持按 provider/service 维度查询，数据持久化到 Redis。

#### 2.1 指标数据模型

#### 2.2 实现方案 — ProviderMetricsCollector

```javascript
// src/modules/tts/infrastructure/ProviderMetricsCollector.js

class ProviderMetricsCollector {
  constructor(redisClient) {
    this._redis = redisClient;
    this._windowSeconds = 3600; // 1小时滑动窗口
  }

  /**
   * 记录一次合成调用
   */
  async record(metric) {
    const key = `provider_metrics:${metric.serviceKey}`;
    const score = Date.now();

    // 写入 Redis sorted set（score = 时间戳）
    await this._redis.zAdd(key, score, JSON.stringify(metric));

    // 设置 TTL（保留 7 天）
    await this._redis.expire(key, 7 * 24 * 3600);

    // 更新聚合统计（原子操作）
    await this._updateAggregates(metric);
  }

  /**
   * 更新实时聚合统计
   */
  async _updateAggregates(metric) {
    const aggKey = `provider_agg:${metric.serviceKey}`;
    const hourKey = new Date().toISOString().slice(0, 13); // "2026-06-07T10"

    await this._redis.hIncrBy(`${aggKey}:${hourKey}`, 'total', 1);
    if (metric.success) {
      await this._redis.hIncrBy(`${aggKey}:${hourKey}`, 'success', 1);
    } else {
      await this._redis.hIncrBy(`${aggKey}:${hourKey}`, 'error', 1);
    }
    await this._redis.hIncrBy(`${aggKey}:${hourKey}`, 'totalLatency', metric.latencyMs);
  }

  /**
   * 查询某 provider 的指标摘要
   */
  async getMetrics(serviceKey, windowMs = 3600000) {
    const key = `provider_metrics:${serviceKey}`;
    const since = Date.now() - windowMs;

    const records = await this._redis.zRangeByScore(key, since, Date.now());
    const parsed = records.map(r => JSON.parse(r));

    if (parsed.length === 0) {
      return { totalCalls: 0, windowMinutes: windowMs / 60000 };
    }

    const latencies = parsed.map(r => r.latencyMs).sort((a, b) => a - b);
    const successCount = parsed.filter(r => r.success).length;

    return {
      totalCalls: parsed.length,
      successRate: (successCount / parsed.length * 100).toFixed(1) + '%',
      latency: {
        p50: latencies[Math.floor(latencies.length * 0.5)],
        p95: latencies[Math.floor(latencies.length * 0.95)],
        p99: latencies[Math.floor(latencies.length * 0.99)],
        avg: Math.round(latencies.reduce((a, b) => a + b, 0) / latencies.length)
      },
      errors: this._groupErrors(parsed.filter(r => !r.success)),
      windowMinutes: windowMs / 60000
    };
  }

  _groupErrors(errorRecords) {
    const grouped = {};
    for (const r of errorRecords) {
      const code = r.errorCode || 'UNKNOWN';
      grouped[code] = (grouped[code] || 0) + 1;
    }
    return grouped;
  }
}

module.exports = { ProviderMetricsCollector };
```

### 方案 3：Provider 自动降级

**目标**：当用户指定的 Provider 不可用时（circuit open / 凭证失效 / 超时），自动选择能力最接近的备选 Provider，用户无感知。

#### 3.1 降级链路设计

```
用户请求 → 指定 minimax_tts
  ↓
minimax_tts circuit closed? → 正常调用 → 返回
  ↓ circuit open
查找备选：
  1. 从 CompiledCapability 能力矩阵中找 "同类" 服务
  2. 按 capabilityDigest 相似度排序
  3. 排除同样 circuit open 的
  4. 选择健康度最高的
  ↓
调用备选 provider → 返回
  response.metadata.degraded = true
  response.metadata.originalProvider = "minimax"
  response.metadata.actualProvider = "moss"
```

#### 3.2 实现 — ProviderFallbackChain

```javascript
// src/modules/tts/infrastructure/ProviderFallbackChain.js

class ProviderFallbackChain {
  constructor({ executionPolicy, capabilityResolver, providerManagement }) {
    this._executionPolicy = executionPolicy;
    this._capabilityResolver = capabilityResolver;
    this._providerManagement = providerManagement;
  }

  /**
   * 获取降级候选列表
   * @param {string} failedServiceKey - 失败的服务 key
   * @returns {string[]} 按优先级排序的备选 serviceKey 列表
   */
  getCandidates(failedServiceKey) {
    const failedCapability = this._capabilityResolver.resolve(failedServiceKey);
    if (!failedCapability) return [];

    const failedDigest = failedCapability.capabilityDigest();
    const allServices = this._providerManagement.getAllServices();

    return allServices
      .filter(s => s.key !== failedServiceKey && s.status !== 'disabled')
      .map(s => {
        try {
          const cap = this._capabilityResolver.resolve(s.key);
          const similarity = this._calculateSimilarity(failedDigest, cap?.capabilityDigest());
          const isHealthy = !this._executionPolicy.isCircuitOpen(s.key);
          return {
            key: s.key,
            similarity,
            healthy: isHealthy,
            score: isHealthy ? similarity : similarity * 0.3
          };
        } catch {
          return { key: s.key, similarity: 0, healthy: false, score: 0 };
        }
      })
      .filter(c => c.score > 0.3)  // 相似度太低的排除
      .sort((a, b) => b.score - a.score)
      .map(c => c.key);
  }

  /**
   * 计算两个能力摘要的相似度（0-1）
   */
  _calculateSimilarity(a, b) {
    if (!a || !b) return 0;

    const keysA = Object.keys(a);
    const keysB = Object.keys(b);
    const allKeys = new Set([...keysA, ...keysB]);

    let matchCount = 0;
    for (const key of allKeys) {
      if (JSON.stringify(a[key]) === JSON.stringify(b[key])) {
        matchCount++;
      }
    }
    return matchCount / allKeys.size;
  }
}
```

#### 3.3 在合成链路中集成降级

```javascript
// TtsSynthesisService.synthesize 中
async synthesize(request) {
  // ... 原有逻辑 ...

  try {
    return await this.executionPolicy.execute(serviceKey, async () => {
      return this._doSynthesize(request, serviceKey, resolvedServiceKey);
    });
  } catch (error) {
    // circuit open 或不可重试的错误 → 尝试降级
    if (this._isDegradable(error)) {
      const candidates = this._fallbackChain.getCandidates(resolvedServiceKey);

      for (const candidateKey of candidates) {
        try {
          // 用候选 provider 重新构建合成链路
          const degradedResult = await this._doSynthesizeWithFallback(
            request, candidateKey
          );
          degradedResult.metadata.degraded = true;
          degradedResult.metadata.originalProvider = identity.providerKey;
          degradedResult.metadata.actualProvider = candidateKey.split('_')[0];
          return degradedResult;
        } catch {
          continue; // 备选也失败，继续尝试下一个
        }
      }
    }

    throw error; // 全部失败，抛出原始错误
  }
}

_isDegradable(error) {
  return ['CIRCUIT_OPEN', 'RATE_LIMIT_EXCEEDED', 'PROVIDER_UNAVAILABLE']
    .includes(error.code);
}
```

```javascript
// TtsQueryService.js 新增方法

matchProviders(requirements = {}) {
  const allServices = this._providerManagement.getAllServices();
  const matched = [];
  const unmatched = [];

  for (const svc of allServices) {
    if (!this._isProviderEnabled(svc.key)) continue;

    const compiled = this._capabilityResolver.resolve(svc.key);
    if (!compiled) continue;

    const digest = compiled.capabilityDigest();
    const matchDetails = {};
    const mismatchReasons = [];
    let score = 0;
    let totalRequirements = 0;

    for (const [reqKey, reqValue] of Object.entries(requirements)) {
      totalRequirements++;

      if (reqKey === 'language') {
        // 语言匹配：检查 supportedLanguages
        const langField = compiled.getField('languageType');
        const supported = langField?.values?.includes(reqValue) ||
                          langField?.values?.includes('Auto');
        matchDetails[reqKey] = !!supported;
        if (supported) score++;
        else mismatchReasons.push(`不支持语言 ${reqValue}`);
      } else if (reqKey === 'format') {
        const formatField = compiled.getField('format');
        const supported = formatField?.values?.includes(reqValue);
        matchDetails[reqKey] = !!supported;
        if (supported) score++;
        else mismatchReasons.push(`不支持格式 ${reqValue}`);
      } else if (reqKey === 'maxLatency') {
        // 延迟要求暂时无法静态判断，跳过
        continue;
      } else {
        // 通用能力匹配：streaming, emotion, etc.
        const field = compiled.getField(reqKey);
        const supported = field && field.status !== 'unsupported';
        matchDetails[reqKey] = !!supported;
        if (supported) score++;
        else mismatchReasons.push(`不支持 ${reqKey} 参数`);
      }
    }

    const scorePercent = totalRequirements > 0
      ? Math.round(score / totalRequirements * 100) : 100;

    const entry = {
      serviceKey: svc.key,
      provider: svc.provider,
      displayName: svc.displayName,
      score: scorePercent,
      matchDetails
    };

    if (scorePercent >= 50) {
      if (mismatchReasons.length) entry.mismatchReasons = mismatchReasons;
      matched.push(entry);
    } else {
      unmatched.push({
        serviceKey: svc.key,
        reason: mismatchReasons.join(', ')
      });
    }
  }

  matched.sort((a, b) => b.score - a.score);

  return { matched, unmatched };
}
```

### 方案 5：Provider 动态热更新

**目标**：修改 manifest.json 后自动热加载，无需重启服务。新增声明式 HTTP Provider 时只需放一个 JSON 文件。

#### 5.1 文件监听 + 重编译

```javascript
// src/modules/tts/config/ManifestWatcher.js

const chokidar = require('chokidar');

class ManifestWatcher {
  constructor({ providerManifest, capabilityCompiler, providerRegistry }) {
    this._manifest = providerManifest;
    this._compiler = capabilityCompiler;
    this._registry = providerRegistry;
    this._watcher = null;
  }

  start(manifestDir) {
    if (this._watcher) return;

    this._watcher = chokidar.watch(
      path.join(manifestDir, '**/*.json'),
      {
        ignoreInitial: true,
        awaitWriteFinish: { stabilityThreshold: 500 }
      }
    );

    this._watcher
      .on('change', (filePath) => this._handleReload(filePath))
      .on('add', (filePath) => this._handleReload(filePath));

    console.log(`[ManifestWatcher] Watching ${manifestDir} for changes`);
  }

  async _handleReload(filePath) {
    try {
      // 清除 require cache
      delete require.cache[require.resolve(filePath)];

      // 重新加载 manifest
      this._manifest.reload();

      // 重新初始化 ProviderRegistry
      this._registry.reinitialize();

      // 清除 CompiledCapability 缓存（触发重编译）
      this._compiler.clearCache();

      console.log(`[ManifestWatcher] Reloaded: ${path.basename(filePath)}`);
    } catch (error) {
      console.error(`[ManifestWatcher] Reload failed: ${error.message}`);
    }
  }

  stop() {
    this._watcher?.close();
    this._watcher = null;
  }
}

module.exports = { ManifestWatcher };
```

## 三、实施优先级

| 优先级 | 方案 | 价值 | 工作量 | 说明 |
|:---:|------|------|:---:|------|
| P0 | 方案 1：消除冗余层 | 代码清晰 | 2-3h | 技术债清理，其他方案的前置条件 |
| P1 | 方案 2：可观测性 | 生产级标志 | 1-2d | 区分"能用"和"专业"的分水岭 |
| P2 | 方案 3：自动降级 | 用户体验 | 2-3d | 用户无感知的容错能力 |
| P3 | 方案 4：能力匹配 | 前端友好 | 1d | 锦上添花，前端不再维护兼容性矩阵 |
| P4 | 方案 5：热更新 | 运维效率 | 1d | 当前体量不紧急，provider 增多后价值上升 |

### 建议实施顺序

```
第 1 周：方案 1（清理）→ 方案 2（可观测性）
第 2 周：方案 3（自动降级）
第 3 周：方案 4 + 方案 5（按需选做）
```

## 四、注意事项

### 4.1 向后兼容

- ProviderCatalog 移除注入后，如果有外部代码依赖 `voiceCatalog` 参数，需要同步修改
- 新增的 metrics 接口建议放 `/api/admin/` 下，需要 admin 权限
- 自动降级的 response 增加了 `degraded` 字段，前端需要适配

### 4.2 测试覆盖

```
方案 2 需要测试：
  - 成功/失败调用各记录一条 metrics
  - Redis 不可用时降级为内存 metrics（不阻塞合成）
  - 滑动窗口过期数据正确清理

方案 3 需要测试：
  - circuit open 时自动降级到备选
  - 所有备选都失败时返回原始错误
  - 降级响应包含正确的 metadata

方案 5 需要测试：
  - manifest 文件修改后自动重编译
  - 无效 JSON 不导致服务崩溃
  - 热更新期间的并发请求不丢数据
```

### 4.3 与能力系统演进方案的配合

本方案与 `能力系统演进方案.md` 中的四个方案相互独立，可以并行推进：

- 能力系统演进方案解决的是**参数编译和映射**的问题
- 本方案解决的是**服务商生命周期管理和运行时质量**的问题
- 方案 2（可观测性）依赖 CompiledCapability 的 capabilityDigest 做能力比对
- 方案 3（自动降级）依赖 CapabilityCompiler 重新编译备选 provider 的能力
