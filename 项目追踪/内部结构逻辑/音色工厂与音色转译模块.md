## CATALOG_AND_REGISTRY_BOUNDARY
- ProviderCatalog:
  role: canonical service resolution, alias mapping, provider capability descriptors.
  path: src/modules/tts/catalog/ProviderCatalog.js
- VoiceCatalog:
  role: static runtime/display voice lookup from dist artifacts.
  path: src/modules/tts/catalog/VoiceCatalog.js
- VoiceRegistry:
  role: dynamic registration/index lifecycle; can diverge from static catalog.
  path: src/modules/tts/core/VoiceRegistry.js
- TtsQueryService:
  role: query aggregation layer for list/filter/frontend payloads.
  path: src/modules/tts/application/TtsQueryService.js
- consequence:
  - listing path and synthesis path can diverge if sources are not explicitly unified.

## BUILD_PIPELINE
- input: provider yaml voice definitions
- transform: normalize schema + derive searchable indexes
- output:
  - voices/dist/voices.json (runtime detail map)
  - voices/dist/categories.json (provider/language/gender/tag indexes)
- invariant:
  - defaultVoiceId in `ttsDefaults` must exist in voices/dist/voices.json

## RUNTIME_TRANSLATION_PIPELINE
- stage_01_normalize:
  file: src/modules/tts/application/VoiceResolver.js
  anchor: src/modules/tts/application/VoiceResolver.js:33
  behavior:
    - merge top-level options + options object
    - unify `voice` / `voiceId`
    - inject default service when missing

- stage_02_service_canonicalize:
  file: src/modules/tts/application/VoiceResolver.js
  anchor: src/modules/tts/application/VoiceResolver.js:59

- stage_03_effective_voice_resolution:
  file: src/modules/tts/application/VoiceResolver.js
  anchor: src/modules/tts/application/VoiceResolver.js:76

    - effectiveVoiceId = explicit voiceId OR ttsDefaults.getDefaultVoiceId(canonicalService)

- stage_04_runtime_voice_mapping:
  file: src/modules/tts/application/VoiceResolver.js
  anchor: src/modules/tts/application/VoiceResolver.js:77
  behavior:
    - VoiceCatalog.getRuntime(effectiveVoiceId)
    - flatten providerOptions into runtime options

- stage_05_options_materialization:
  file: src/modules/tts/application/VoiceResolver.js
  anchors:
    - _normalizeVoiceRuntime: src/modules/tts/application/VoiceResolver.js:94
    - _buildRuntimeOptions: src/modules/tts/application/VoiceResolver.js:118
  behavior:
    - merge defaults + runtime voice + request options
    - ensure both voice and voiceId exist
    - fallback to `default` sentinel if absent

## DEFAULT_FALLBACK_RULES
priority_order:
  1_explicit_request: request.voiceId or request.voice
  2_service_default: ttsDefaults.byService[canonicalService].defaultVoiceId
  3_sentinel: voice='default' and voiceId='default' (resolver final fallback)

## FAILURE_MODES
- F1_default_voice_drift:
  symptom: fallback voice invalid / missing
  check:
    - src/modules/tts/config/ttsDefaults.js
    - voices/dist/voices.json

- F2_alias_canonical_gap:
  symptom: shortcut service works partially but wrong default voice/adapter
  check:
    - route-injected service alias
    - ProviderCatalog canonical mapping

- F3_catalog_duality:
  symptom: management voice list differs from synthesis runtime behavior
  check:
    - VoiceCatalog vs any legacy registry path usage

## QUICK_VALIDATION_PROTOCOL
- V1: verify defaultVoiceId existence in dist catalog for each service
- V2: submit synth request without voice; assert stable fallback by service
- V3: submit alias service values; assert canonical adapter selection
- V4: compare /voices listing path and synth runtime voice mapping consistency
