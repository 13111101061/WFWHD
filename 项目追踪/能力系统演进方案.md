# TTS 能力系统演进方案

## 一、现状概述

当前能力系统由三层构成：

```
platform-fields.json (13 个平台基线字段，硬编码)
        ↓
FieldDefinitionRegistry — 加载基线 + 读取 manifest 覆盖
        ↓
CapabilityCompiler — 三层合并编译（基线 + manifest 覆盖 + provider 映射）
        ↓
CompiledCapability — 运行时只读对象（schema / defaults / locked / validate / resolveParams / mapToProvider）
```

### 合成执行链路

```
TtsSynthesisService.synthesize()
  → capabilityResolver.resolve(serviceKey)          // 获取缓存的 CapabilityContext
  → parameterResolutionService.mergeFromContext()   // 委托 compiled.resolveParams()
  → compiled.validate()                             // 同源校验
  → compiled.mapToProvider()                        // 平台参数 → 服务商 API 参数
  → adapter.synthesize(providerParams)              // 发往服务商
```

### 当前服务商清单

| 服务商 | Service Key | 凭证状态 | Adapter 类型 | supported / unsupported / locked |
|--------|-------------|----------|-------------|----------------------------------|
| 阿里云 | `aliyun_cosyvoice` | ✅ 已配置 | 自定义 Adapter | 7 / 5 / 1 |
| 阿里云 | `aliyun_qwen_http` | ✅ 已配置 | 自定义 Adapter | 5 / 7 / 1 |
| MOSS | `moss_tts` | ✅ 已配置 | Generic HTTP | 5 / 6 / 2 |
| MiMo | `mimo_tts` | ✅ 已配置 | 自定义 Adapter | 4 / 7 / 2 |
| MiniMax | `minimax_tts` | ❌ 未配置 | Generic HTTP | 8 / 4 / 1 |
| 腾讯云 | `tencent_tts` | ❌ 未配置 | 自定义 Adapter | 5 / 7 / 1 |
| 火山引擎 | `volcengine_http` | ❌ 未配置 | 自定义 Adapter | 4 / 8 / 1 |

### 现有设计优势（保留不动）

1. **三层合并编译** — platform 基线 + manifest 覆盖 + provider 映射，职责清晰
2. **CompiledCapability 运行时只读** — 启动时编译、零运行时开销
3. **capabilityDigest 校验** — 前端提交 digest 比对，不一致返回 `CAPABILITY_SCHEMA_OUTDATED`
4. **锁定参数透明注入** — voice/model 通过 `applyLockedParams()` 自动注入，用户无感
5. **mapTo 映射体系** — direct/rename/linear/enumMap/nestedPath 五种 transform，覆盖所有服务商差异
6. **resolveParams 单一入口** — 过滤 → 合并默认 → 锁定注入 → warnings，一条链路走完

## 二、已识别问题

### P0：platform-fields.json 字段硬编码

**文件位置**：`src/modules/tts/config/fields/platform-fields.json`

**问题**：13 个字段写死在单个 JSON 文件中。新增字段（如某个服务商独有的 `reverb` 混响参数）需要：
1. 改 `platform-fields.json`（影响全局）
2. 重启服务重新编译
3. 所有服务商的编译结果都会多一个 `unsupported` 字段

**当前 13 个字段**：
```
voice, text, instruction, speed, pitch, volume,
format, sampleRate, emotion, expectedDurationSec,
samplingParams, languageType, model
```

**实际影响**：`expectedDurationSec` 只有 MOSS 用，`samplingParams` 只有 MOSS 用，`instruction` 只有 MiMo 用。但这些字段被写在了平台基线里，其他服务商编译后都会出现 `unsupported` 条目。

### P1：缺少服务商独有参数扩展通道

**当前做法**：manifest 只能对 platform-fields.json 中**已有的字段**做覆盖（status/default/range/values/mapTo）。如果服务商需要一个全新的参数，没有干净的扩展方式。

**案例**：如果某个服务商需要 `noiseSuppression`（降噪等级），当前只能：
- 改 platform-fields.json 加字段 → 污染其他 6 个服务商
- 或走 `providerOptions` 绕过 → 丧失 schema 校验和 UI 渲染

### P2：缺少轻量字段依赖表达

**案例**：MiMo 的 `instruction` 与 `speed/pitch/volume/emotion` 互斥。当前通过前端逻辑处理（看到 `instruction` 是 supported 就隐藏其他控件），没有在 schema 层面声明这种关系。

### P3：unsupported 参数静默丢弃

**当前行为**：`CompiledCapability.resolveParams()` 对 unsupported 参数加 warning 后直接丢弃。

**期望行为**：某些场景下 unsupported 参数可以降级转换。例如用户传了 `speed=1.5` 但服务商不支持 speed，可以自动转为 `instruction` 里追加 "语速: 1.5x"。

## 三、改进方案

### 方案 A：字段文件插件化（解决 P0）

**改动范围**：`FieldDefinitionRegistry._loadPlatformFields()`

**做法**：

1. 将 `platform-fields.json` 拆分为 `fields/` 目录下的独立文件：

```
src/modules/tts/config/fields/
  voice.field.json
  text.field.json
  instruction.field.json
  speed.field.json
  pitch.field.json
  volume.field.json
  format.field.json
  sampleRate.field.json
  emotion.field.json
  languageType.field.json
  model.field.json
  samplingParams.field.json       ← 含 nestedFields
  expectedDurationSec.field.json  ← MOSS 专用，后续迁移到 manifest
```

2. `FieldDefinitionRegistry` 启动时扫描 `fields/*.field.json`，合并为 `platformFields`

3. 保留 `platform-fields.json` 中的 `uiGroups` 和 `categories` 定义（可放在 `_meta.json` 中）

**单字段文件格式**：

```json
{
  "$field": "speed",
  "key": "speed",
  "displayName": "语速",
  "description": "语音播放速度，1.0 为正常速度",
  "type": "number",
  "category": "audio",
  "required": false,
  "platformDefault": 1.0,
  "platformRange": { "min": 0.25, "max": 4.0 },
  "ui": {
    "group": "audio",
    "order": 1,
    "inputType": "slider",
    "step": 0.1,
    "labels": { "0.5": "慢", "1.0": "正常", "2.0": "快" }
  },
  "validation": { "range": { "min": 0.25, "max": 4.0 } }
}
```

**核心改动**（约 30 行）：

```javascript
// FieldDefinitionRegistry._loadPlatformFields()
_loadPlatformFields() {
  const fs = require('fs');
  const path = require('path');
  const fieldsDir = path.join(__dirname, 'fields');
  const entries = fs.readdirSync(fieldsDir, { withFileTypes: true });

  this._platformFields = { fields: {}, uiGroups: {}, categories: {} };

  for (const entry of entries) {
    if (!entry.isFile() || !entry.name.endsWith('.field.json')) continue;
    const content = JSON.parse(fs.readFileSync(path.join(fieldsDir, entry.name), 'utf8'));
    const fieldKey = content.$field || content.key;
    if (fieldKey) this._platformFields.fields[fieldKey] = content;
  }

  // 加载 _meta.json (uiGroups + categories)
  const metaPath = path.join(fieldsDir, '_meta.json');
  if (fs.existsSync(metaPath)) {
    const meta = JSON.parse(fs.readFileSync(metaPath, 'utf8'));
    this._platformFields.uiGroups = meta.uiGroups || {};
    this._platformFields.categories = meta.categories || {};
  }
}
```

**向后兼容**：保留旧的 `platform-fields.json` 作为 fallback，如果 `fields/` 目录不存在则走旧逻辑。

### 方案 B：manifest customFields 支持（解决 P1）

**改动范围**：`ProviderManifest.getFieldOverrides()` + `CapabilityCompiler.compile()`

**做法**：在 manifest.json 中新增 `customFields` 块，声明服务商独有参数：

```json
{
  "providerKey": "moss",
  "services": {
    "moss_tts": {
      "parameters": { ... },
      "customFields": {
        "expectedDurationSec": {
          "displayName": "期望时长",
          "description": "期望生成的语音时长（秒）",
          "type": "number",
          "default": 5,
          "range": [0.5, 300],
          "mapTo": "expected_duration_sec",
          "ui": { "group": "advanced", "order": 1, "inputType": "number", "unit": "秒" }
        }
      }
    }
  }
}
```

**编译器改动**（约 20 行）：

```javascript
// CapabilityCompiler.compile() — 在遍历 platformFields 之后追加：
const customFields = ProviderManifest.getCustomFields(serviceKey) || {};
for (const [fieldKey, customDef] of Object.entries(customFields)) {
  if (compiledFields[fieldKey]) continue; // 不覆盖平台字段

  const compiled = compileField(fieldKey, customDef, null, customDef.mapTo ? {
    providerPath: customDef.mapTo,
    transform: 'direct'
  } : null);

  compiled.validator = generateValidator(compiled);
  compiled.mapper = generateMapper(compiled, apiStructure);
  compiledFields[fieldKey] = compiled;

  if (fieldIndex[compiled.status]) fieldIndex[compiled.status].push(fieldKey);
}
```

**迁移路径**：
- 将 `expectedDurationSec` 从 platform-fields.json 迁移到 moss/manifest.json 的 `customFields`
- 将 `samplingParams` 从 platform-fields.json 迁移到 moss/manifest.json 的 `customFields`
- 将 `instruction` 从 platform-fields.json 迁移到 mimo/manifest.json 的 `customFields`
- platform-fields.json 只保留真正的全局通用字段（约 10 个）

### 方案 C：轻量 dependsOn 依赖（解决 P2）

**改动范围**：`platform-fields.json` 字段定义 + `CompiledCapability.getSchema()` 输出

**做法**：在字段定义中加 `dependsOn` 声明：

```json
{
  "$field": "emotion",
  "dependsOn": {
    "field": "instruction",
    "condition": "absent"
  }
}
```

含义：当 `instruction` 字段状态为 `supported` 时，`emotion` 自动变为 `hidden`。

**支持的 condition 类型**：

| condition | 含义 |
|-----------|------|
| `"absent"` | 依赖字段不存在或不 supported 时才显示 |
| `"present"` | 依赖字段 supported 时才显示 |
| `"valueEquals": "xxx"` | 依赖字段值等于某值时才显示 |

**编译器处理**（约 15 行）：编译时把 `dependsOn` 写入 compiledField，前端从 schema 中读取并做条件渲染。

```javascript
// compileField() 中追加：
if (platformField?.dependsOn) {
  compiled.dependsOn = platformField.dependsOn;
}
```

**前端消费示例**：

```javascript
// 前端渲染逻辑
for (const [key, field] of Object.entries(schema)) {
  if (field.dependsOn) {
    const depField = schema[field.dependsOn.field];
    if (field.dependsOn.condition === 'absent' && depField?.status === 'supported') {
      continue; // 隐藏此字段
    }
  }
  renderField(key, field);
}
```

### 方案 D：unsupported 降级策略（解决 P3）

**改动范围**：`CompiledCapability.resolveParams()` + manifest 字段定义

**做法**：manifest 中为 unsupported 字段声明 fallback：

```json
"speed": {
  "status": "unsupported",
  "reason": "通过 instruction 自然语言控制",
  "fallback": {
    "target": "instruction",
    "template": "语速调整为 ${value} 倍速"
  }
}
```

**resolveParams 改动**（约 15 行）：

```javascript
// CompiledCapability.resolveParams() — 在 UNSUPPORTED 分支中：
if (status === SupportStatus.UNSUPPORTED) {
  const field = this.getField(key);

  if (field?.fallback?.target && field.fallback.template) {
    const fallbackText = field.fallback.template.replace(/\$\{value\}/g, String(value));
    const targetKey = field.fallback.target;
    result[targetKey] = result[targetKey]
      ? `${result[targetKey]}。${fallbackText}`
      : fallbackText;
  }

  warnings.push({
    type: 'unsupported',
    param: key,
    value,
    message: field?.reason || `此服务商不支持参数 ${key}`,
    fallbackApplied: !!field?.fallback
  });
  continue;
}
```

## 四、实施优先级

| 优先级 | 方案 | 改动量 | 影响面 | 收益 |
|--------|------|--------|--------|------|
| **P0** | A. 字段文件插件化 | ~30 行 + 文件拆分 | FieldDefinitionRegistry | 新字段不改现有文件 |
| **P0** | B. customFields 支持 | ~20 行 + manifest 改动 | CapabilityCompiler + manifest | 服务商独有参数有干净出口 |
| P1 | C. dependsOn 依赖 | ~15 行 | 编译器 + 前端 | schema 层声明互斥 |
| P1 | D. 降级策略 | ~15 行 | CompiledCapability | 参数自动降级不丢失 |

### 建议实施顺序

```
第一周:  A（字段插件化）→ 验证所有端点不受影响
第二周:  B（customFields）→ 迁移 expectedDurationSec / samplingParams / instruction
第三周:  C + D（依赖 + 降级）
```

## 五、不建议做的

| 方向 | 原因 |
|------|------|
| 全面迁移 JSON Schema 标准 | 前端是自写渲染逻辑，不是 @rjsf/core；JSON Schema 的 oneOf/anyOf/$ref 用不到，徒增复杂度 |
| 运行时能力发现 | 服务商是本地 manifest 文件，不是远程注册中心；启动编译 + reload() 热重载已够用 |
| 推倒重做 | 三层合并编译 + CompiledCapability 只读对象 + digest 校验，这套设计在 TTS 聚合领域是领先的 |
| 参数组合约束引擎 | 当前只有 MiMo 的 instruction 互斥一个场景，dependsOn 足以覆盖，不需要完整规则引擎 |
